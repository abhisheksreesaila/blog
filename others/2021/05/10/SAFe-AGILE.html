Introducing Scrum in SAFe

  • User Stories

    • Small unit of value
    • it has to be user driven
    • I am driver, wanting to do this, that’s adds "this" business value
    • It needs to have
      • CARDS
      • CONVERSATION
      • CONFIRMATION
    • Personas as Representative User

  • Features

    • bigger chunk of user stories
    • single value stream of work
    • typically achieved under a quarter

  • Capability

    • multiple streams of work

  • Velocity

    • The sum of the all the story points that are completed by the SCRUM team in the iteration
    • If they finish 50% of a story, DO NOT take 50% of story point to calculate velocity
    • Its either DONE or NOT DONE

  • Epics

    • not same as projects
    • work of substantial value to the organization
    • can run into mulitple PI's
    • does not have start and end time
    • Define MVP clearly
    • Try to achieve them, if you cannot, pivot and rewrite epics
    • Refer below diagram

  • Architecture Runaway

    • Achieve the near-term features without excessive redesign and delays
    • It will spike up and down
    • Imagine an aircraft, needing to land after exit

  • Enablers

    • Architectural Enabler

      • Improve Perf
      • Becomes Non-Functional-Requirement (NFR) after implementation

    • Infra Enabler

      • CI/CD

  • Difference Between SAFe and SCRUM Terminology

Important: Cadence and Synchronization are the key constructs to delivering continuous value

Characterizing the role of the Scrum Master

  • Role of a SCRUM Master

    • Coach the team with AGILE SCRUM process
    • Facilitate problem solving
    • Protect them from external influence of stakeholders
    • Represent in SOS (Scrum of Scrums)
    • Helps in Coordination with other team (but responsibility of everyone in the AGILE team)
    • Acts as SERVANT leader
      • The one who knows that his/her own growth comes from facilitating the growth of others
    • Ask Powerful questions
      • surfaces hidden assumptions

  • SCRUM of SCRUMS (SOS) = twice a week, then cut down

  • RTE - Release train engineer

    • Manager for scrum masters

  • Shift from old behaviors like ===> new behaviors

    • Driving towards an answer ==> become a facilitator
    • Knowing the answer ===> Letting the team find their own way
    • Directing ===> Guiding

Experiencing PI Planning

  • PI Planning

    • Top 10 features as input
    • PI objectives
    • or else do not start

  • User-less stores

    • DB ==> front end

  • Time is not EQUAL story points

    • relative
    • pick a base and measure against it

  • Never start sprints on Monday/and end on fridays

  • PI planning

    • <>iteration planning
    • assign to people. (atleast team)
    • atleast 2 stories so that we hit the ground running

IP (innovation planning) iteration

  • Innovation
  • No testing
  • No Bug fixes
  • No System Integration

If we dont do this...

  • Tech debt grows
  • People burn out
  • No time to improve
    Warning: Anti-Patterns in Estimations
    - Pressure by stakeholders to loqwer estimations
    • Only few people participate
    • NOt using adjusted fibo scale
      Warning: Anti-Patterns for Breakout Sessions
      - Too much time spent on each story
    • No plan at the end of the timebox
      Warning: Anti-Patterns for PI Planning
      - Pressure on teams to overcommit
    • Team under commits due to over failure
    • Over-planning ahead of time

Facilitating Iteration Execution

  • Iteration Planning
    • consists of 4 steps (PDCA: Plan, Do, Check, Adjust)
      • Iteration Planning
      • Iteration Execution
      • Iteration Retrospective
      • Iteration Review
        Warning: Anti-Patterns for Iteration Progress
        - No input from SoS
        • Teams unwilling to edit objectives mid-PI

  • IP Iteration (Special Type)

    • Innovation purposes
    • NO testing
    • NO bug fixes
    • NO system integration
    • If we dont this iteration, then tech debt grows
    • Burn out among people

  • Daily stand ups

    • 15 mins
    • use the next 15 min for stay-behind
      Warning: Anti-Patterns for Iteration review
      - Focusing on only what to improve and not what to preserve

Finishing the PI

  • System demo is given to the stakeholders (May include multiple demo)

  • Quantatitive measurement (Actual vs Planned)

  • Problem solving workshop (Retro at a high level)

    Warning: Anti-Patterns
    - Only PO presents in the system demo
  • No actionable feature improvements created
  • Improvements are not shown
